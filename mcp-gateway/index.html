The problem

Your agents run on borrowed trust

Most teams wire AI agents into internal tools with long-lived API keys, wildcard scopes, and zero auditability. When something goes wrong — and it will — you can't answer the one question that matters: which agent did what, when, and with whose credentials?

⚠

Over-provisioned tokens

Agents get broad-scope API keys because scoping per-tool is manual, tedious, and fragile. One compromised agent means lateral movement across every service it can reach.

⚡

Zero agent-level audit

Logs say "the API key did X." But which agent? Which model? Which invocation? Without agent-scoped identity, your SIEM can't tell the difference between your code-review bot and your production deployer.

🔒

No per-call policy enforcement

Even if you scope a token, it's static. You can't say "this agent may read from S3 during business hours, but only from the staging bucket, and never after 3 consecutive failures."

How it works

Three pieces. Five minutes. No YAML.

The Gateway sits between your agents and your tools. It's not a proxy — it's an identity broker. You register tools once, then every call is authenticated, authorized, and logged.

1

Register your tools and MCP servers

Point the Gateway at your existing tools — APIs, databases, MCP servers, internal services. Each becomes a registered resource with its own policy surface.

$ gateway register --name "staging-db" --type postgres --host db.staging.internal ✓ Registered resource res_8xK2mP9v with default deny-all policy
2

Define agents with scoped identity

Every agent — whether it's Claude Code, a custom MCP client, or a CI pipeline — gets its own identity with short-lived credentials, scoped to exactly the tools it needs, for exactly the operations it should perform.

$ gateway agent create --name "code-review-bot" \ --allow res_8xK2mP9v:read \ --allow res_3aLn7Qw4:read \ --max-ttl 15m ✓ Agent agt_xY9vR3kL created. Credentials expire in 15m.
3

Every call is attested

The Gateway issues a per-call attestation token. Every tool invocation produces a tamper-evident log entry — agent identity, tool, operation, timestamp, and a cryptographic chain that proves the entry hasn't been altered. Ship logs to your SIEM or keep them in the Gateway.

$ gateway logs --agent agt_xY9vR3kL --last 5 2026-06-25T14:03:11Z agt_xY9vR3kL READ res_8xK2mP9v sig:0x7f3a... 2026-06-25T14:03:09Z agt_xY9vR3kL READ res_3aLn7Qw4 sig:0xa91c...

Features

Built for security teams, not for demos

Every feature exists because a team running agents at production scale asked for it. No AI-generated filler.

Short-lived credentials

No static API keys. Credentials auto-rotate and expire. Default TTL is 5 minutes. You set the ceiling.

Per-call policy engine

Rego-based policies evaluated on every invocation. Rate limits, time windows, parameter constraints, chained approvals.

Tamper-evident audit log

Every entry is hashed and chained. You can verify the integrity of the entire log with a single command. Exports to Splunk, Datadog, or S3.

MCP-native

First-class support for the Model Context Protocol. Register MCP servers as resources. Agents connect through the Gateway without code changes.

Agent SDK (3 lines)

Drop-in SDKs for Python, TypeScript, and Go. Replace your bare API calls with Gateway-authenticated calls in minutes.

IdP integration

Wire agents to your existing identity provider — Okta, Azure AD, or any OIDC provider. Agent identity maps to your org's directory.

Team workspaces

Separate environments for dev, staging, and production. Each workspace has its own resources, agents, and policies.

CLI-first, API-everything

Every operation available via CLI and REST API. Terraform provider for GitOps workflows. No UI-lock-in.
